Docker za Windows Početnike

Kompletan vodič: od objašnjenja pojmova do naprednih komandi.

1. Šta je zapravo Docker?

Zamislimo kako se roba nekada prevozila brodovima: u vrećama, kutijama raznih oblika, buradima... Bilo je nevjerovatno teško sve to složiti na brod, a još teže prebaciti na kamion. Onda je izmišljen brodski kontejner — standardna metalna kutija. Bez obzira šta je unutra (televizori, kafa, automobili), svaki kran i svaki brod na svijetu tačno zna kako da podigne i prenese taj kontejner.

Docker je upravo to, ali za softver.

Prije Dockera, proces pokretanja tuđeg koda izgledao je ovako: "Da bi moj program radio kod tebe, moraš instalirati tačno Python verziju 3.10, pa instalirati ovu bazu podataka, pa podesiti ove varijable...". To je često rezultiralo poznatom frazom: "Ali kod mene na računaru radi!".

Sa Dockerom, programer zapakuje svoj kod, zajedno sa svim programima koji mu trebaju (Python, baza, biblioteke) u jedan virtuelni "kontejner". Vi samo preuzmete taj kontejner i pokrenete ga. Vaš računar ne mora imati ništa instalirano (ni Python ni bazu) — sve je zatvoreno u tom kontejneru. Ako radi na programerovom kompjuteru, garantovano će raditi i na vašem.

Kontejneri vs. Virtuelne Mašine (VM)

Možda ste čuli za VirtualBox ili VMware (virtuelne mašine). Virtuelna mašina (VM) u sebi sadrži cijeli operativni sistem (npr. instalira cijeli Windows ili Linux unutar vašeg Windowsa). To je jako tromo, sporo se pali i troši mnogo RAM-a.

Docker kontejneri nemaju svoj operativni sistem. Oni posuđuju jezgro vašeg sistema. Zato su kontejneri teški samo nekoliko megabajta, pale se za par sekundi i ne troše gotovo nimalo resursa kada ne rade ništa.

Dva najvažnija pojma u Dockeru:
  • Image (Slika): Ovo je "recept" ili "nacrt". To je fajl koji sadrži sve instrukcije. Ne radi ništa, samo stoji na disku (kao instalacijski fajl igre).
  • Container (Kontejner): Ovo je "živa" aplikacija. Nastaje onog trenutka kada pokrenete Image (kao kada zapravo igrate igru). Od jednog Image-a možete pokrenuti 100 istih Kontejnera.

2. Instalacija i Deinstalacija na Windows-u

Instalacija

  1. Otiđite na službenu Docker stranicu i preuzmite Docker Desktop for Windows.
  2. Pokrenite instalaciju. Pojavit će se opcija "Use WSL 2 instead of Hyper-V". OBAVEZNO ostavite ovo označeno. (WSL 2 je "Windows Subsystem for Linux" — fantastičan Microsoftov alat koji omogućava Dockeru da radi brzo na Windows-u).
  3. Nakon instalacije, Windows će tražiti restart računara.
  4. Nakon restarta, pokrenite aplikaciju Docker Desktop (pronađite je u Start meniju).
  5. Sačekajte dok ikonica u donjem lijevom uglu (ili dole u taskbaru) ne postane zelena. Tek tada je Docker spreman za rad.

Provjera da li sve radi: Otvorite PowerShell (ili Command Prompt) i upišite:

docker run hello-world

Ako vidite poruku "Hello from Docker!", instalacija je savršena.

Deinstalacija

Ako nekada želite potpuno ukloniti Docker:

  1. Idite na Add or remove programs u Windows postavkama.
  2. Pronađite Docker Desktop i kliknite Uninstall.
  3. Da biste obrisali zaostale ogromne fajlove koje je Docker ostavio (npr. slike kontejnera), izbrišite folder: C:\Users\VašeImeKorisnika\AppData\Local\Docker.

3. Trajnost podataka (Docker Volumes)

Zlatno pravilo Dockera: Kontejneri su POTROŠNI (ephemeral). Ako obrišete kontejner, SVI podaci koji su bili unutar njega zauvijek nestaju!

Zamislimo da ste pokrenuli kontejner za bazu podataka (npr. SQL Server). Vaša aplikacija je cijeli dan spremala podatke u bazu. Ugasite računar, Docker ubije kontejner, sutra ga upalite — baza je potpuno prazna! Zašto? Jer su podaci bili zapisani "unutar" kontejnera.

Da bismo ovo spriječili, koristimo Volumes (Volumene). Volume je prozorčić u zidu kontejnera kroz koji kontejner sprema podatke direktno na vaš pravi hard disk (Windows).

Postoje dva načina čuvanja podataka:

1. Bind Mounts (Mapiranje vašeg foldera)

Kažete Dockeru: "Umiješto da podatke spremaš kod sebe, spremaj ih u moj folder C:\moji_podaci".

docker run -v "C:\moji_podaci:/putanja/u/kontejneru" ime_imagea

2. Docker Volumes (Preporučeno)

Kažete Dockeru: "Kreiraj ti jedan poseban zaštićeni prostor na mom disku pod nazivom sql_podaci, i tu čuvaj podatke nezavisno od kontejnera".

# Prvo kreiramo volumen
docker volume create sql_podaci

# Zatim pokrećemo kontejner i spajamo ga na volumen
docker run -v sql_podaci:/var/opt/mssql ime_imagea

Kada sutra obrišete kontejner i pokrenete novi, samo ga ponovo spojite na volumen sql_podaci i vaši podaci su tu!

4. Osnovne Docker komande

Komanda Šta radi?
docker ps Prikazuje listu kontejnera koji trenutno RADE.
docker ps -a Prikazuje listu SVIH kontejnera (i onih koji rade i onih koji su ugašeni/pali).
docker pull <image> Samo preuzima Image (nacrt) sa interneta na vaš računar. Ne pokreće ga. (npr. docker pull ubuntu)
docker run <image> Pokreće kontejner iz datog image-a. (Često se koriste dodaci poput -d da radi u pozadini, ili -p za portove).
docker stop <id> Zaustavlja upaljeni kontejner, ali ga NE briše sa računara. Podaci u njemu i dalje postoje.
docker rm <id> Potpuno briše kontejner! (Gubite sve podatke koji nisu sačuvani u Volumen).
docker logs <id> Prikazuje šta se ispisivalo u terminalu tog kontejnera (odlično za pronalazak grešaka ako aplikacija ne radi).
docker exec -it <id> bash "Ulazi" unutar kontejnera, dajući vam terminal kako biste mogli kucati komande kao da ste na tom sistemu.

5. Šta je docker-compose.yml?

Zamislimo da naša aplikacija (poput onih na edukaciji) treba:

  • SQL Bazu podataka
  • Ollama server za AI modele
  • ChromaDB vektorsku bazu

Da bismo ovo pokrenuli koristeći obične Docker komande, morali bismo kucati tri ogromne linije koda, svaka sa po 10 parametara (za portove, lozinke, volumene). Ako pogriješimo jedno slovo, ne radi. I to moramo raditi svaki dan.

Docker Compose to rješava. On nam omogućava da sva ta pravila zapišemo u jedan tekstualni fajl (docker-compose.yml). Onda u terminalu ukucamo samo jednu jedinu komandu:

docker compose up -d

I on automatski pročita fajl i pokrene sve tri aplikacije povezane u mrežu!

Analiza stvarnog docker-compose.yml fajla

Ovo je fajl koji se nalazi u root folderu vaših projekata (x:\Projects\GitHub\AI-Training\docker-compose.yml). Razbijmo ga na dijelove:

version: '3.9' # Verzija sintakse koju koristimo

services:      # Ovdje počinjemo listati naše kontejnere

  # Prvi kontejner: SQL Server
  sqlserver:
    image: mcr.microsoft.com/mssql/server:2022-latest # Koji image skidamo s interneta
    container_name: ai-trening-sqlserver             # Ime koje ćemo vidjeti u docker ps
    environment:                                     # Lozinke i postavke koje aplikacija traži
      ACCEPT_EULA: "Y"
      SA_PASSWORD: "Training123!"
    ports:
      - "1433:1433"   # Prespajamo port 1433 našeg računara na port 1433 kontejnera
    volumes:
      - sqlserver_data:/var/opt/mssql # Bazu podataka spašavamo trajno u volumen!

  # Drugi kontejner: Ollama AI server
  ollama:
    image: ollama/ollama:latest
    container_name: ai-trening-ollama
    ports:
      - "11434:11434"
    volumes:
      - ollama_models:/root/.ollama   # AI Modele teške gigabajte spašavamo trajno

# Ovdje na kraju fajla govorimo Dockeru koje Volumene da zaista kreira na našem hard disku
volumes:
  sqlserver_data:    
  ollama_models:     
  chroma_data:
Najvažnije Compose komande:
  • docker compose up -d — Pokreće sve sisteme opisane u yml fajlu u pozadini (-d znači detached).
  • docker compose down — Zaustavlja i briše kontejnere, ali ČUVA volumene (podaci su sigurni).
  • docker compose down -v — Brisanje svega UKLJUČUJUĆI i volumene. Oprez, ovo nepovratno briše bazu i skinute AI modele!

6. Pokretanje pojedinih servisa — pristup za edukaciju

Na ovom treningu koristimo više servisa (SQL Server, Ollama, ChromaDB), ali nije ih potrebno pokretati sve odjednom. Svaki dan edukacije koristi samo određene servise. Pokretanje samo onoga što trebate štedi RAM, CPU i smanjuje konfuziju.

Organizacija docker/ foldera

Sve Docker Compose konfiguracije nalaze se u folderu docker/. Svaki servis ima vlastiti podfolder s docker-compose.yml unutar njega. Iz tog foldera pokrenete standardnu komandu — bez ikakvih dodatnih parametara:

AI-Training/
└── docker/
    ├── README.md          ← Kratko uputstvo
    ├── ollama/            ← SAMO Ollama (Dan 1, Dan 3)
    │   └── docker-compose.yml
    ├── sql/               ← SAMO SQL Server
    │   └── docker-compose.yml
    ├── chroma/            ← SAMO ChromaDB
    │   └── docker-compose.yml
    ├── no-sql/            ← Ollama + ChromaDB (Dan 2, Dan 4)
    │   └── docker-compose.yml
    └── full/              ← Svi servisi zajedno
        └── docker-compose.yml

Koji servis, za koji dan i zašto

Dan / Projekat Potrebni servisi Folder za pokretanje Zašto baš taj servis?
Dan 1 — Tokeni & Temperatura Ollama (opcionalno) cd docker/ollama Aplikacija može raditi s DeepSeek API-jem bez Ollame. Ollama je fallback za offline rad.
Dan 2 — Film Chat ChromaDB + Ollama cd docker/no-sql ChromaDB čuva embeddinze filmova. Ollama ili cloud API odgovaraju na pitanja.
Dan 3 — Ollama projekat Ollama cd docker/ollama Dan je posvećen isključivo lokalnim AI modelima. SQL i ChromaDB nisu potrebni.
Dan 4 — Prognoza SQL Server + Ollama cd docker/no-sql Podaci o vremenskim prognozama čuvaju se u SQL bazi.
Kompletan setup Svi servisi cd docker/full Za predavača ili napredne korisnike koji rade više projekata istovremeno.

Kako pokrenuti pojedinačni servis

Uđite u odgovarajući folder i pokrenite standardnu komandu — bez ikakvih dodatnih parametara:

# Dan 1 i Dan 3 — samo Ollama
cd docker/ollama
docker compose up -d

# Dan 2 — Ollama + ChromaDB
cd docker/no-sql
docker compose up -d

# SQL Server
cd docker/sql
docker compose up -d

# Samo ChromaDB
cd docker/chroma
docker compose up -d

# Sve odjednom (predavački računar)
cd docker/full
docker compose up -d

Provjera i zaustavljanje

# Pregled svih aktivnih kontejnera (iz bilo kojeg foldera)
docker ps

# Praćenje logova uživo (iz foldera gdje ste pokrenuli)
docker compose logs -f

# Zaustavljanje — iz ISTOG foldera odakle ste pokrenuli
docker compose down

# Zaustavljanje + brisanje volumena (PAŽNJA: nepovratno!)
docker compose down -v
Savjet za predavače: Na početku svakog dana edukacije, uđite u odgovarajući folder i pokrenite samo servis koji taj dan treba. Studenti tada vide samo jedan kontejner u docker ps listi — manje konfuzije, lakše praćenje. Naredni dan: docker compose down u prethodnom folderu, cd u novi, docker compose up -d.

7. Ollama — upravljanje lokalnim AI modelima

Ollama je posebno važna komponenta ovog treninga jer nam omogućava da pokrećemo AI modele potpuno lokalno — bez interneta, bez plaćanja, bez slanja podataka na cloud. Razumijevanje kako Ollama radi ključno je za Dan 3 i sve projekte koji ne koriste cloud API.

Analogija: Ollama kao "garaža za AI modele"

Zamislite Ollamu kao garažu, a AI modele kao automobile:

  • Garaža (Ollama) je uvijek na istom mjestu, uvijek dostupna na portu 11434
  • Automobili (modeli: Llama, Mistral, Phi...) su različite veličine i namjena
  • Možete imati više automobila u garaži — ali vozite samo jedan u datom trenutku (ili nekoliko, uz dijeljenje resursa)
  • Kada trebate drugi model, jednostavno ga "izvučete iz garaže" — bez reinstalacije

Kako Ollama prima zahtjeve od aplikacija

Kada vaša Python aplikacija pošalje pitanje AI-u, tok izgleda ovako:

Vaša Python aplikacija
        ↓
HTTP zahtjev na http://localhost:11434/v1/chat/completions
        ↓
Ollama server (Docker kontejner) prima zahtjev
        ↓
Ollama učita model (npr. llama3.2:1b) u RAM
        ↓
Model generira odgovor token po token
        ↓
Ollama vraća odgovor vašoj aplikaciji

Preuzimanje AI modela

Kad Ollama kontejner radi, modeli se preuzimaju komandom ollama pull. Postoje dva načina:

# Način 1: Direktno unutar kontejnera
docker exec ai-trening-ollama ollama pull llama3.2:1b

# Način 2: Ako imate Ollamu instaliranu i lokalno (ne u Dockeru)
ollama pull llama3.2:1b
Važno — veličine modela: Modeli su veliki fajlovi koji se preuzimaju jednom i čuvaju u Docker volumenu ollama_models. Preuzimanje traje nekoliko minuta ovisno o brzini interneta i veličini modela.

Upravljanje modelima — pregled komandi

# Preuzimanje modela
docker exec ai-trening-ollama ollama pull llama3.2:1b

# Lista svih preuzetih modela (s veličinom i datumom)
docker exec ai-trening-ollama ollama list

# Provjera koji modeli trenutno RADE (učitani u RAM)
docker exec ai-trening-ollama ollama ps

# Testiranje modela direktno u terminalu (interaktivni chat)
docker exec -it ai-trening-ollama ollama run llama3.2:1b

# Brisanje modela (oslobađa prostor na disku)
docker exec ai-trening-ollama ollama rm llama3.2:1b

# Informacije o modelu
docker exec ai-trening-ollama ollama show llama3.2:1b

Chat sa modelom iz Terminala (bez koda)

Kao što je prikazano u komandama iznad, možete pokrenuti interaktivni chat sa modelom direktno u vašem terminalu (PowerShell, Command Prompt ili WSL).

Pokrenite sljedeću komandu: 

docker exec -it ai-trening-ollama ollama run llama3.2:1b

Napomena: Ako imate Ollamu instaliranu lokalno na računaru (nativno, van Docker-a), dovoljno je pokrenuti samo: 

ollama run llama3.2:1b

Nakon što se chat pokrene, možete kucati pitanja i komunicirati sa modelom. Kada želite izaći iz chat sesije, upišite /bye i pritisnite Enter, ili pritisnite Ctrl + D.

Mogu li dva AI modela raditi istovremeno?

Odgovor: Da — ali uz važan asterisk.

Ollama može imati više modela učitanih istovremeno, ali to dijeli isti RAM i CPU. Evo šta se zapravo dešava:

  • Model se učita u RAM kada ga prvi put pozovete. Ostaje učitan neko vrijeme (zadano: 5 minuta neaktivnosti).
  • Ako pozovete drugi model dok je prvi još u RAMu, Ollama može oba držati u memoriji — ako imate dovoljno RAM-a.
  • Na računarima s manje RAM-a (npr. 8 GB), Ollama će automatski istisnuti prethodni model kada treba učitati novi.
  • Zahtjevi se obrađuju sekvencijalno po modelu — Ollama nije dizajnirana za paralelne zahtjeve na isti model od više korisnika.
Praktično pravilo za edukaciju: Na jednom studentskom laptopu (8-16 GB RAM), koristite samo jedan model istovremeno. Prebacivanje između modela radi promjenom OLLAMA_MODEL_SMALL u .env fajlu i restartom aplikacije — ne trebate uopšte dirati Docker.

Mijenjanje aktivnog modela za aplikaciju

Da promijenite koji model vaša aplikacija koristi, ne trebate mijenjati kod. Dovoljno je:

  1. Preuzeti željeni model: docker exec ai-trening-ollama ollama pull ime-modela
  2. Otvoriti .env fajl u projektu
  3. Promijeniti vrijednost OLLAMA_MODEL_SMALL=novi-model
  4. Restartovati Python server (Ctrl+C, pa ponovo uvicorn)

Aplikacija automatski pokupi novu vrijednost iz .env fajla pri svakom startu.

8. OLLAMA_MODEL_SMALL i MEDIUM u .env fajlu

Svaki projekat u ovom treningu koristi .env fajl za konfiguraciju. U tom fajlu postoje dvije varijable za Ollama modele: OLLAMA_MODEL_SMALL i OLLAMA_MODEL_MEDIUM. Ova sekcija objašnjava šta su, kada se koji koristi i kako ih mijenjati.

Šta piše u .env fajlu

# Ollama konfiguracija (lokalni LLM server)
OLLAMA_BASE_URL=http://localhost:11434
OLLAMA_MODEL_SMALL=llama3.2:1b
OLLAMA_MODEL_MEDIUM=llama3.2:3b

Ovo su samo tekstualne varijable — same po sebi ne rade ništa. Vrijednost dobivaju tek kada ih Python kod eksplicitno pročita i preda Ollami.

Šta znači "Small" a šta "Medium"?

Varijabla Podrazumijevana vrijednost Veličina modela Tipična namjena
OLLAMA_MODEL_SMALL llama3.2:1b 1.3 milijardu parametara (~700 MB) Brzi odgovori, demo, testiranje. Radi čak i na starijim laptopovima s 4 GB RAM-a.
OLLAMA_MODEL_MEDIUM llama3.2:3b 3.2 milijarde parametara (~2 GB) Bolji kvalitet odgovora. Preporučeno kada imate 8+ GB RAM-a i malo više strpljenja.

Kada aplikacija koristi SMALL, a kada MEDIUM?

To ovisi o kodu svake aplikacije. Svaki projekat čita onu varijablu koja mu je najprikladnija. Na primjer, u Dan 1 projektu (main.py):

# Iz main.py — Dan 1 projekat
# Aplikacija traži OLLAMA_MODEL_SMALL jer je Dan 1 demo projekat
# koji treba brze odgovore, ne nužno visok kvalitet

model = os.getenv("OLLAMA_MODEL_SMALL", "llama3.2:1b")
#                  ↑ varijabla iz .env   ↑ default ako varijabla ne postoji

Aplikacija uvijek čita OLLAMA_MODEL_SMALL ako je to ono što joj treba za određenu funkciju. Možete imati aplikaciju koja koristi SMALL za brze klasifikacije, a MEDIUM za detaljne analize — sve u istom kodu, sve konfigurirano kroz .env.

Kako promijeniti model bez promjene koda

Ovo je ključna prednost .env pristupa. Cijeli postupak:

  1. Otvorite .env fajl u VS Code-u
  2. Promijenite vrijednost, npr: OLLAMA_MODEL_SMALL=mistral:7b
  3. Provjerite da je model preuzet: docker exec ai-trening-ollama ollama list
  4. Restartujte server (Ctrl+C u terminalu, pa ponovo pokrenite uvicorn)
  5. Gotovo — aplikacija sad koristi Mistral umjesto Llame, a kod se nije promijenio ni za slovo
Česta greška: Promijenite .env fajl, ali ne restartujete server. .env se čita samo pri startu aplikacije — promjene ne stupaju na snagu dok ne restartujete. Ovo je namjerno ponašanje: production aplikacije ne smiju dinamički mijenjati konfiguraciju u toku rada.

Može li aplikacija koristiti model koji nije preuzet?

Ne — i Ollama će vratiti grešku. Tok je sljedeći:

# Scenarij: u .env stoji OLLAMA_MODEL_SMALL=llama3.1:8b
# ali taj model NIJE preuzet

# Aplikacija pošalje zahtjev Ollami → Ollama odgovori greškom:
# "model 'llama3.1:8b' not found, try 'ollama pull llama3.1:8b'"

# Rješenje: preuzmite model PRIJE pokretanja aplikacije
docker exec ai-trening-ollama ollama pull llama3.1:8b

Praktični workflow za predavače

  1. Jutro prije nastave: Pokrenite Ollama kontejner i preuzmite modele koji će se koristiti taj dan
  2. Studenti kopiraju .env.example → .env: Podrazumijevane vrijednosti (llama3.2:1b) rade odmah
  3. Napredni studenti: Mogu promijeniti na llama3.2:3b ako žele boji kvalitet
  4. Eksperiment na času: Zajedno promijenite model u .env, restartujte, usporedite odgovore

9. Čišćenje diska — brisanje modela, images i volumena

Tokom rada sa Dockerom i Ollamom, na disku se akumulira značajna količina podataka: AI modeli mogu biti teški po nekoliko gigabajta, a Docker image-i, zaustavljeni kontejneri i stari volumeni polako "jedu" prostor. Ova sekcija objašnjava kako to sve pregledati i očistiti — korak po korak.

Prije brisanja — provjeri šta imaš! Uvijek prvo pokrenite komande za pregled (npr. ollama list, docker images) da vidite šta postoji, pa tek onda brišite. Brisanje je nepovratno.

Korak 1 — Provjera koliko prostora Docker i Ollama zauzimaju

Prije nego počnete brisati, dobro je znati koliko prostora zapravo zauzimate:

# Ukupan pregled zauzetog prostora od strane Dockera
# (images, kontejneri, volumeni, build cache)
docker system df

# Detaljnija verzija — prikazuje svaki resurs posebno
docker system df -v

Tipičan ispis izgleda otprilike ovako:

TYPE            TOTAL     ACTIVE    SIZE      RECLAIMABLE
Images          5         2         8.2GB     5.1GB (62%)
Containers      3         1         12kB      12kB (100%)
Local Volumes   3         2         4.7GB     0B (0%)
Build Cache     12        0         1.3GB     1.3GB

Kolona RECLAIMABLE pokazuje koliko prostora možete osloboditi brisanjem neaktivnih resursa.

# Provjera slobodnog prostora na disku (Windows PowerShell)
Get-PSDrive C | Select-Object Used, Free

Korak 2 — Brisanje pojedinačnih Ollama modela

Ollama modeli se čuvaju unutar Docker volumena ollama_models. Brišu se komandom ollama rm izvršenom unutar kontejnera:

# Prvo — pregledaj koji modeli su preuzeti (s veličinom)
docker exec ai-trening-ollama ollama list

# Primjer ispisa:
# NAME              ID            SIZE    MODIFIED
# llama3.2:1b       ...           1.3 GB  2 hours ago
# llama3.2:3b       ...           2.0 GB  3 days ago
# mistral:7b        ...           4.1 GB  1 week ago
# phi3:mini         ...           2.3 GB  2 weeks ago
# Brisanje jednog modela
docker exec ai-trening-ollama ollama rm llama3.2:1b

# Brisanje više modela odjednom (svaki posebno)
docker exec ai-trening-ollama ollama rm mistral:7b
docker exec ai-trening-ollama ollama rm phi3:mini

# Provjera nakon brisanja
docker exec ai-trening-ollama ollama list
Savjet: Nakon brisanja modela, ne morate restartovati Ollamu ni Docker. Model je odmah uklonjen. Ako ga ponovo trebate, jednostavno ga ponovo preuzmite s ollama pull ime-modela.

Korak 3 — Brisanje Docker image-a (slika)

Docker image-i su "nacrti" kontejnera. Akumuliraju se na disku svaki put kada preuzmete novi ili buildujete vlastiti. Možete ih slobodno brisati — kontejner koji iz njih radi neće biti pogođen dok ga ne restartujete.

# Pregled svih preuzetih image-a (s veličinom)
docker images

# Primjer ispisa:
# REPOSITORY                        TAG       IMAGE ID       SIZE
# ollama/ollama                     latest    abc123...      1.5GB
# mcr.microsoft.com/mssql/server    2022-...  def456...      2.9GB
# chromadb/chroma                   latest    ghi789...      620MB
# Brisanje jednog image-a (koristite IMAGE ID ili naziv:tag)
docker rmi ollama/ollama:latest

# Ako image koristi neki kontejner, Docker će javiti grešku.
# Prvo zaustavite i obrišite kontejner:
docker stop ai-trening-ollama
docker rm ai-trening-ollama
docker rmi ollama/ollama:latest

# Brisanje SVIH image-a koji se trenutno ne koriste ni od jednog kontejnera
docker image prune -a
Pažnja uz docker image prune -a: Ova komanda briše sve image-e koji nemaju pokrenuti kontejner. Sljedeći put kad pokrenete docker compose up, Docker će morati ponovo preuzeti image-e s interneta (može potrajati nekoliko minuta, ovisno o brzini veze).

Korak 4 — Brisanje Docker volumena

Volumeni čuvaju trajne podatke (AI modele, baze podataka, vektore). Brisanje volumena je najopasnija operacija — podaci su nepovratno izgubljeni.

# Pregled svih volumena i njihove veličine
docker volume ls

# Detaljni pregled (s veličinom) — zahtijeva malo više vremena
docker system df -v | grep -A 20 "Local Volumes"
# Brisanje jednog volumena (SAMO ako kontejner koji ga koristi ne radi!)
docker volume rm ollama_models

# Brisanje svih volumena koji NISU prikačeni ni za jedan kontejner
docker volume prune

# Brisanje volumena zajedno s kontejnerima (iz foldera docker-compose-a)
cd docker/ollama
docker compose down -v   # Zaustavlja kontejnere I briše njihove volumene
UPOZORENJE — docker compose down -v: Ovo briše i sve preuzete AI modele iz ollama_models volumena! Ako imate modele teške po 4–8 GB, morat ćete ih ponovo preuzimati. Koristite samo ako ste sigurni da vam ti podaci više ne trebaju.

Korak 5 — Brisanje zaustavljenih kontejnera

Kontejneri koji su zaustavljeni (status: Exited) i dalje zauzimaju prostor, mada obično malen. Vrijedi ih povremeno počistiti:

# Pregled SVIH kontejnera (i aktivnih i zaustavljenih)
docker ps -a

# Brisanje jednog zaustavljenog kontejnera
docker rm ai-trening-ollama

# Brisanje SVIH zaustavljenih kontejnera odjednom
docker container prune

Korak 6 — "Nuklearno čišćenje" (sve odjednom)

Ako želite osloboditi što više prostora odjednom i početi od nule, postoji jedna komanda koja briše sve nekorištene resurse:

# Briše sve odjednom: zaustavljene kontejnere, nekorištene image-e,
# nekorištene mreže i build cache.
# NE briše volumene!
docker system prune -a

# Isto, ali UKLJUČUJE i volumene (MAKSIMALNO agresivno — sve nestaje!)
docker system prune -a --volumes
Ekstremna pažnja uz docker system prune -a --volumes:
Ova komanda briše bukvalno sve što Docker ima — sve image-e, sve volumene, sve cache. Koristite samo kada stvarno želite početi od apsolutne nule i ne smeta vam što ćete sve ponovo preuzimati.

Preporučeni redosljed čišćenja (za studente)

Ako vam ponestaje prostora na disku, pratite ovaj redosljed — od najsigurnijeg prema agresivnijem:

  1. Obrišite Ollama modele koje ne koristite — najsigurnije, modeli se lako ponovo preuzmu: 
    docker exec ai-trening-ollama ollama list
docker exec ai-trening-ollama ollama rm ime-modela
  2. Očistite build cache i zaustavljene kontejnere — nema gubitka podataka: 
    docker system prune
  3. Obrišite nekorištene image-e — Docker će ih ponovo preuzeti po potrebi: 
    docker image prune -a
  4. Tek kao posljednji korak — brisanje volumena (podaci su izgubljeni): 
    docker compose down -v   # iz odgovarajućeg docker/ foldera

Sažetak — šta koja komanda briše i koliko je opasna

Komanda Šta se briše Podaci izgubljeni? Opasnost
ollama rm <model> Jedan AI model iz volumena Model (ponovo se preuzme) 🟡 Niska
docker container prune Svi zaustavljeni kontejneri Ne (podaci u volumenima ostaju) 🟢 Bezopasno
docker image prune -a Svi nekorišteni image-i Ne (image-i se ponovo preuzmu) 🟡 Niska
docker system prune -a Kontejneri + image-i + cache Ne (volumeni ostaju) 🟡 Srednja
docker volume rm <volumen> Jedan volumen (baza, modeli...) DA — nepovratno! 🔴 Visoka
docker compose down -v Kontejneri + svi vezani volumeni DA — nepovratno! 🔴 Visoka
docker system prune -a --volumes Apsolutno sve što Docker ima DA — sve! 🔴🔴 Maksimalna
Pro tip za predavače: Na kraju svakog dana edukacije, sigurno je pokrenuti docker system prune (bez -a i bez --volumes) — to čisti samo privremeni cache i zaustavljene kontejnere, a ne dira AI modele ni baze. Studenti sutra nastavljaju bez problema.
