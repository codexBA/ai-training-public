Deo 1 — Teorija function callinga

Od chatbota do agenta

Do sada smo LLM posmatrali kao izuzetno pametnu, ali izolovanu „enciklopediju". Mogli ste mu postaviti pitanje i on bi generisao tekst. Međutim, šta ako korisnik pita: "Gdje radi Emin?" ili "Koji je status mog poreskog predmeta broj 12345?"

Standardni LLM to ne može znati — nema pristup vašoj bazi ni internim API-jima. Kada bi model pokušao direktno odgovoriti, najvjerovatnije bi halucinirao (npr. izmislio posao ili ustanovu).

Function calling (ili Tool Use) je ključna razlika između običnog chatbota i AI agenta — model može zatražiti da aplikacija izvrši funkciju i vrati mu stvarne podatke.

Industrijski nazivi: OpenAI zove ovo Function Calling. Anthropic koristi Tool Use. U kodu često vidite tools, tool_calls, tool_choice. Sve se odnosi na isti obrazac: model bira alat, aplikacija izvršava.

Analogija: šef i asistent

Zamislimo da je LLM šef zatvoren u sobi bez interneta, a vaša Python aplikacija asistent s telefonom i pristupom bazama.

  1. Korisnik pita šefa: "Gdje radi Emin?"
  2. Šef zna da to ne zna iz glave. Okreće se asistentu: "Iskoristi alat lookup_person za ime Emin." (ovo je tool_call).
  3. Asistent (vaš Python kod) poziva HTTP API, dohvaća JSON i vraća šefu: "Emin Hadžić, softverski inženjer, FZZA Sarajevo."
  4. Šef (LLM) oblikuje lijepu rečenicu korisniku: "Emin Hadžić radi kao softverski inženjer u FZZA Sarajevo."
Ključna zabluda: LLM ne izvršava kod! LLM je jezični model — prima i vraća tekst. Kad želi alat, generiše JSON: "Pozovi funkciju X s argumentom Y". Na vama je backend petlja koja taj JSON pročita, izvrši HTTP/SQL u aplikaciji i pošalje rezultat nazad modelu.

Kako model odlučuje da li treba alat?

Model ne koristi magiju — koristi pattern matching na tekstu pitanja i upoređuje ga s description poljem u JSON schema svakog alata.

Pitanje korisnikaModel odlučujeRazlog
"Gdje radi Emin?"Poziva lookup_person"radi" + ime → match s opisom alata
"Koga sve poznaješ?"Poziva list_people"koga" + lista → drugi alat
"Koliko je 2+2?"Odgovara direktno: "4"Nema alata za matematiku
"Šta je Python?"Odgovara direktnoOpšte znanje — nijedan alat nije relevantan
"Zdravo, kako si?"Odgovara direktnoNema veze s adresarom
Zašto je description KRITIČAN: description u JSON Schema je zapravo prompt za alat. Loš opis ("Dohvata podatke") = model ne zna KADA ga koristiti. Dobar opis: "Dohvata podatke o osobi iz adresara. Koristi SAMO kada korisnik pita gdje neko radi, živi ili čime se bavi."

Zašto ne staviti JSON direktno u prompt?

Mogli biste u system prompt ubaciti cijeli adresar od 10 osoba. Za mali primjer bi radilo, ali:

  • Tokeni — veliki adresar troši kontekst i košta više
  • Zastarjelost — promjena u bazi zahtijeva ažuriranje prompta
  • Sigurnost — model vidi sve odjednom; alati omogućavaju kontrolu pristupa
  • Skalabilnost — u produkciji imate milione redova; alat dohvaća samo što treba

Function calling je industrijski standard za povezivanje LLM-a s live podacima — CRM, eUprava, interni API-ji.

JSON Schema — kako modelu objašnjavamo alate

Da bi model znao koje alate ima, moramo mu to precizno opisati kroz JSON Schema. Taj JSON šaljemo uz svaki upit kroz parametar tools. 

{
  "type": "function",
  "function": {
    "name": "lookup_person",
    "description": "Dohvata podatke o osobi iz adresara. Koristi SAMO kada korisnik pita gdje neko radi, živi ili čime se bavi.",
    "parameters": {
      "type": "object",
      "properties": {
        "name": {
          "type": "string",
          "description": "Ime ili puno ime osobe, npr. Emin"
        }
      },
      "required": ["name"]
    }
  }
}

U mini-adresar projektu ova definicija živi u listi TOOLS u main.py — detaljno u Koraku 5.

Paralelni tool pozivi

Kada korisnik pita za više stavki odjednom (npr. "Uporedi gdje rade Emin i Amra"), model može vratiti više tool_calls u jednom koraku:

SEKVENCIJALNO vs PARALELNO SEKVENCIJALNO: Iteracija 1: tool_calls = [lookup_person("Emin")] Iteracija 2: tool_calls = [lookup_person("Amra")] Iteracija 3: LLM kombinira → finalni odgovor PARALELNO (DeepSeek/GPT-4o): Iteracija 1: tool_calls = [lookup_person("Emin"), lookup_person("Amra")] → Oba se izvršavaju → Oba rezultata Iteracija 2: LLM kombinira → finalni odgovor

Naš run_tool_loop podržava paralelne pozive jer iterira kroz cijelu listu msg.tool_calls.

Graceful degradation: Kada alat vrati {"greska": "..."}, ta poruka ide nazad LLM-u kao role: tool. Model formira pristojan odgovor: "Osoba nije u adresaru" — korisnik ne vidi stack trace.

Primjena i sigurnost — Human-in-the-Loop

Adresar je pokazni primjer, ali stvarna moć function callinga u javnom sektoru leži u integraciji s postojećim sistemima. Agent nije zamjena za softver — pametni interfejs na vrhu softvera.

AlatOpisTip
provjeri_status_predmeta(broj)SELECT u SQL baziRead (sigurno)
izracunaj_porez(...)Precizna matematika po formulamaRead (sigurno)
obrisi_predmet(id)DELETE u baziWrite (OPASNO)
posalji_email(...)Slanje službenog emailaWrite (OPASNO)
Human-in-the-Loop: Read operacije su uglavnom sigurne. Write/Update/Delete nikada ne smiju biti potpuno prepušteni modelu — model može pogriješiti argumente (npr. id=55 umjesto id=5). Destruktivni alat treba pripremiti akciju i tražiti odobrenje čovjeka u UI.
HUMAN IN THE LOOP — destruktivne operacije BEZ HITL (OPASNO): Korisnik → LLM → tool_call: obrisi(id=55) → SQL DELETE SA HITL (SIGURNO): Korisnik → LLM → tool_call: pripremi_brisanje(id=55) → UI: "AI predlaže brisanje #55. [Odobri] [Odbij]" → Čovjek provjerava prije izvršenja

Deo 2 — Mini Adresar (praksa i kod)

Teorijski uvod — Zašto ovaj projekat i šta gradimo?

Mini Adresar je edukativni projekat koji demonstrira function calling — mehanizam gdje AI model ne odgovara samo tekstom, već može zatražiti da aplikacija izvrši funkciju (alat) i vrati mu rezultat. Korisnik pita prirodnim jezikom: "Gdje radi Emin?", a sistem:

  1. Prepozna da treba podatak iz adresara
  2. Pozove alat lookup_person s argumentom {"name": "Emin"}
  3. Alat pošalje HTTP GET na simulirani API
  4. Rezultat se vrati modelu kao role: tool poruka
  5. Model generiše finalni odgovor na bosanskom

Zašto radimo ovaj projekat i čemu služi?

U dan4-prognoza radite isti function calling obrazac s pravim vanjskim API-jem (Open-Meteo), cache-om i HTTP integracijom. Mini Adresar skraćuje koncept na minimum — lokalni JSON adresar, simulirani HTTP servis unutar iste aplikacije, bez interneta. Oba projekta koriste stateless /api/ask (bez chat historije na serveru).

Cilj je da jasno vidite granicu odgovornosti: LLM odlučuje KADA pozvati alat, ali vaš Python kod izvršava alat i vraća rezultat. Model nikad ne čita fajl data/korisnici.json direktno — vidi samo ono što alat vrati preko HTTP-a.

Šta problem rješava function calling?

Problem bez alataRješenje s function callingom
Model izmišlja podatke (halucinacije) — npr. kaže da Emin radi u Googleu Model poziva lookup_person i odgovara samo na osnovu stvarnog JSON odgovora
Model nema pristup vašoj bazi, fajlovima ili internim API-jima Vi definišete alate; model šalje strukturirani zahtjev, vi izvršavate
Korisnik mora znati SQL, REST API ili komande Korisnik pita prirodnim jezikom; AI prevodi u poziv alata
Podaci se mijenjaju (novi zaposlenik u adresaru) Alat uvijek čita aktuelne podatke — model ne mora biti re-treniran

Finalna kontrolna lista:
  • Možete objasniti 5 koraka function calling petlje kolegi
  • Razumijete zašto LLM ne izvršava kod sam (analogija šef/asistent)
  • Možete napisati JSON Schema za novi alat s name, description, parameters, required
  • Znate razliku Read vs Write operacija i zašto je Human-in-the-Loop obavezan
  • Razumijete zašto LLM ne čita korisnici.json direktno
  • Znate razliku između sim-API-ja, alata i run_tool_loop
  • Testirali ste sim-API s lookup_direct.py prije AI testa
  • Vidjeli ste [TOOL CALL] u CLI agentu ili tool_trace na webu
  • Razumijete stateless chat — svaki /api/ask je nezavisan
