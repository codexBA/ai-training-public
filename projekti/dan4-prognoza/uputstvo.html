Teorijski uvod — Zašto ovaj projekat i šta gradimo?

Dobrodošli u Dan 4! U ovom projektu gradimo AI asistenta za vremensku prognozu koji ne izmišlja temperature — već poziva prave Python funkcije koje dohvaćaju podatke s Open-Meteo API-ja.

Zašto radimo ovaj projekat i čemu služi?

U Dan 2 ste naučili Chat API i system prompt. U Dan 3 ste radili s Ollamom lokalno. Sada dolazi sljedeći nivo: model ne mora znati sve iz glave — može tražiti da aplikacija izvrši funkciju i vrati mu rezultat. To je osnova modernih AI agenata, chatbota s alatima i asistenata koji pretražuju web, baze podataka ili interne sisteme.

Šta želimo demonstrirati?

  • Function Calling (Tool Use): Model prima JSON opis alata i odlučuje kada i koji alat pozvati, s kojim argumentima.
  • Izvršenje alata u Pythonu: LLM ne izvršava kod — mi u execute_tool() pokrećemo pravu funkciju i vraćamo JSON rezultat modelu.
  • Tool loop petlja: Vi pitate → model traži alat → vi izvršite → model čita rezultat → model odgovara korisniku na bosanskom.
  • Vanjski API bez ključa: Open-Meteo je besplatan — demonstriramo integraciju stvarnih podataka s cache-om i timeout-om.
  • Trace / vidljivost: Frontend i CLI pokazuju šta je AI pozvao — ključno za učenje i debug.

Tri puta do vremenskih podataka

Projekat ima tri načina da stignete do istih Open-Meteo podataka. Razlikuju se po tome koristite li AI i function calling:

PutKada koristitiŠta učite
cli/meteo_direct.py ili GET /api/weather/{city} Test API-ja, debug, bez LLM troška httpx, cache, JSON struktura prognoze
POST /api/ask (web ili curl) Korisnik piše prirodni jezik Function calling, tool_trace
run_tool_loop unutrašnje poruke Automatski — ne dirate ručno OpenAI protokol (tool_calls, role: tool)
PUT 1 — Bez AI (direktno) meteo_direct.py ili GET /api/weather/Mostar │ ▼ get_weather_forecast() → Open-Meteo API PUT 2 — AI + alati (stateless) POST /api/ask { "pitanje": "Kakva je prognoza za Mostar?" } │ ▼ LLM bira alat → execute_tool() → get_weather_forecast() → Open-Meteo │ ▼ LLM formatira odgovor na bosanskom + tool_trace u JSON-u
Stateless vs tool-loop poruke — ne miješajte ova dva pojma!
  • Stateless — svaki POST /api/ask šalje samo 2 poruke (system + jedan user). Server ne pamti prethodna pitanja. Web i CLI rade ovako.
  • Tool-loop poruke — privremene poruke unutar jednog zahtjeva dok LLM ne završi (assistant s tool_calls, zatim role: tool s rezultatom). To nije chat historija iz Dana 2 — nema višeturnskog razgovora.

Analogija: Stateless = svaki put novi razgovor s operaterom. Tool-loop = operater telefonira kolegi (alat), čeka podatke, pa vam odgovori — sve u jednom pozivu.

Predavač: Zamislite call centar. Korisnik pita operateru (AI): "Kakva je prognoza za Mostar?" Operater ne zna napamet — ali ima dugme "Provjeri prognozu" (alat).         city = args.get("city", "")

def get_llm_client() -> tuple[AsyncOpenAI, str]:

async def run_tool_loop(klijent, model, messages, tool_choice="auto"):
    trace = []
    current_messages = list(messages)

    for iteration in range(MAX_TOOL_ITERATIONS):
        response = await klijent.chat.completions.create(
            model=model,
            messages=current_messages,
            tools=TOOLS,
            tool_choice=tool_choice if iteration == 0 else "auto",
            temperature=0.3,
            max_tokens=500,
        )
        msg = response.choices[0].message

        if not msg.tool_calls:
            return msg.content or "", trace, iteration + 1

        current_messages.append(msg.model_dump(exclude_none=True))

        for tc in msg.tool_calls:
            fn_name = tc.function.name
            fn_args = tc.function.arguments or "{}"
            result_str = execute_tool(fn_name, fn_args)
            trace.append(ToolTraceEntry(...))
            current_messages.append({
                "role": "tool",
                "tool_call_id": tc.id,
                "content": result_str,
            })

    return "Ne mogu dovršiti zahtjev — previše koraka alata.", trace, MAX_TOOL_ITERATIONS
Korak petljeŠta se dešava
1. LLM pozivŠaljemo messages + tools=TOOLS
2. Nema tool_calls?Model je spreman odgovoriti — vraćamo tekst
3. Ima tool_callsDodajemo assistant poruku u privremenu listu poruka (samo za ovaj zahtjev)
4. Za svaki alatexecute_tool() → JSON string rezultat
5. Tool porukarole: "tool" + tool_call_id — model zna koji rezultat pripada kojem pozivu
6. PonoviModel čita rezultate i ili poziva još alata ili odgovara
TOOL LOOP — jedna iteracija messages: [system, user] │ ▼ LLM + tools=TOOLS │ ├─► nema tool_calls ──► return odgovor korisniku │ └─► ima tool_calls │ ├─► execute_tool("get_weather_forecast", '{"city":"Mostar"}') ├─► append role:tool + rezultat JSON └─► sljedeća iteracija petlje

async def ask(zahtjev: AskRequest):

    Stateless pitanje — šalje samo system + user poruku (nema historije).
